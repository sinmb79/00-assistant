# 22B Assistant — Project Tree v5 (Final)

## Product Identity

**22B Assistant**

A local-first AI assistant with a team of specialized agents.
Each agent is an expert in a specific domain.
They don't chat with each other — they each do their job and report back.

---

## Core Principles

1. **Local-only by default**: all data stays on user's PC
2. **Agent = specialist**: each agent has domain expertise (rules + corpus + role)
3. **Pipeline, not conversation**: agents don't debate — they work and report
4. **Security-first**: Security Agent monitors all 4 gates of every pipeline
5. **Hybrid LLM**: user chooses local (free) or external (own account)
6. **Progressive**: works without LLM, better with it, best with external LLM

## Pipeline Architecture

```
사용자 요청
    │
    ▼
┌─────────────────────────────────────────────────┐
│ 🔒 GATE 1: Input Classification                 │
│    보안감사관이 입력 내용의 민감도 판단            │
│    → 민감 등급 태깅 (공개/내부/대외비/비밀)       │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│ 🎯 COORDINATOR: Task Decomposition              │
│    요청 분석 → 필요한 에이전트 선택 → 업무 분배    │
│    예: "착공신고서 검토해줘"                       │
│    → 토목 에이전트: 기술 요건 체크                 │
│    → 행정 에이전트: 공문서 형식 체크               │
│    → 인허가 에이전트: 인허가 요건 체크             │
└──┬──────┬──────┬──────┬─────────────────────────┘
   │      │      │      │
   ▼      ▼      ▼      ▼    (병렬 실행 가능)
┌──────┐┌──────┐┌──────┐┌──────┐
│Agent ││Agent ││Agent ││ ...  │
│  A   ││  B   ││  C   ││      │
│      ││      ││      ││      │
│입력→ ││입력→ ││입력→ ││      │
│처리→ ││처리→ ││처리→ ││      │
│결과  ││결과  ││결과  ││      │
└──┬───┘└──┬───┘└──┬───┘└──┬───┘
   │       │       │       │
   ▼       ▼       ▼       ▼
┌─────────────────────────────────────────────────┐
│ 🔒 GATE 2: External Transmission Guard           │
│    외부 LLM 호출이 필요한 에이전트가 있으면:       │
│    → 마스킹 처리                                 │
│    → 사용자 확인 ("이대로 전송하시겠습니까?")      │
│    → 승인 시 전송, 결과 수신 후 역치환            │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│ 🔒 GATE 3: Result Verification                   │
│    각 에이전트 결과물 검증:                        │
│    → 인용된 법령 조문이 실제 DB에 존재하는가        │
│    → 교정 제안이 규칙 DB에 근거가 있는가           │
│    → 결과물에 민감정보가 노출되지 않았는가          │
│    → 에이전트 간 결과가 상충하지 않는가            │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│ 🎯 COORDINATOR: Result Assembly                  │
│    검증된 결과들을 하나의 응답으로 조합             │
│    출처/근거 명시                                 │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────┐
│ 🔒 GATE 4: Audit Log                            │
│    전체 파이프라인 기록:                           │
│    → 어떤 요청이 들어왔는가                       │
│    → 어떤 에이전트가 투입되었는가                  │
│    → 외부 전송이 있었는가 (마스킹 내역 포함)       │
│    → 최종 결과는 무엇인가                         │
│    → 사용자가 수용/거절했는가                      │
└────────────────────┬────────────────────────────┘
                     │
                     ▼
                사용자에게 전달
```

## Agent Roster

### Standing Agents (상시 근무)

#### 🎯 Coordinator (코디네이터)
```
역할: 업무 접수 → 분석 → 에이전트 선택 → 분배 → 결과 조합
LLM:  로컬 (의도 분류에 문서 내용 불포함)
상태: 항상 활성

판단 기준:
  키워드 + 패턴 매칭 (Phase A)
  → 로컬 LLM 의도 분류 (Phase C)

예시:
  "이 공문 교정해줘" → [행정]
  "착공신고서 검토해줘" → [토목, 행정, 인허가]
  "내일까지 할 일 정리" → [일정]
  "수의계약 기준금액?" → [법무, 회계]
```

#### 🔒 Security Auditor (보안감사관)
```
역할: 전 파이프라인 보안 감시 (Gate 1~4)
LLM:  항상 로컬 (보안 판단 자체가 외부 불가)
상태: 항상 활성

Gate 1 — 입력 분류:
  정규식 + NER로 민감정보 스캔
  민감 등급 태깅: 공개 / 내부 / 대외비 / 비밀
  등급에 따라 LLM 라우팅 제한

Gate 2 — 외부 전송 차단:
  Presidio 기반 PII 마스킹
  마스킹 결과 사용자 확인 UI
  승인 시에만 외부 전송
  역치환

Gate 3 — 결과 검증:
  환각 감지 (인용 조문 DB 대조)
  결과물 내 민감정보 노출 체크
  에이전트 간 결과 상충 감지

Gate 4 — 감사 로그:
  전 과정 기록 (암호화 SQLite)
  누가, 언제, 무엇을, 어떤 경로로 처리했는지
  외부 전송 기록 (마스킹 전후 매핑 포함)
```

### Domain Agents (전문 에이전트 — 필요 시 투입)

#### 📝 Administrative Agent (행정 에이전트)
```
역할: 공문서 교정, 작성, 형식 검토, 행정 절차 안내
코어: 구 P1 공문닥터 엔진
지식: 행정업무운영편람, 공문서 작성 요령, 맞춤법 규칙
LLM:  규칙 교정은 로컬, 윤문은 사용자 설정에 따라

업무:
  - 맞춤법/띄어쓰기 교정 (규칙 엔진)
  - 공문서체 교정 (규칙 엔진)
  - 문장 윤문 (LLM)
  - 기안문/보고서/회의록 초안 작성 (LLM)
  - 행정 절차/서식 안내 (규칙 + 코퍼스)
```

#### ⚖️ Legal Agent (법무 에이전트)
```
역할: 법령 검색, 조문 해석, 적용 법령 찾기
코어: 구 P2 규정레이다 엔진
지식: 전체 현행법령 코퍼스, 법령해석례, 판례
LLM:  검색은 로컬 엔진, 설명은 사용자 설정에 따라

업무:
  - 자연어 법령 검색 (BM25 + Vector)
  - 관련 조문 인용 + 이유 설명
  - 법령 비교 (신구 대조)
  - 법령 개정 알림
```

#### 🏗️ Civil Engineering Agent (토목 에이전트)
```
역할: 토목 분야 기술 기준, 수량 산출, 공법 판단
지식: 건설기술진흥법 체계, 토목 설계기준, 표준시방서
LLM:  하이브리드

업무:
  - 공사 규모별 적용 기준 안내
  - 수량/공사비 개략 산정
  - 착공/준공 서류 체크리스트
  - 설계변경 요건 검토
  - 품질/안전 관리 기준 안내
```

#### 🏢 Architecture Agent (건축 에이전트)
```
역할: 건축법 체계, 건축 심의, 설계 기준
지식: 건축법 체계, 건축 구조기준, 에너지절약설계기준
LLM:  하이브리드

업무:
  - 건축허가 요건 검토
  - 건축 심의 대상 판단
  - 용도변경/대수선 기준
  - 건축법 해석례 검색
```

#### 📋 Permit Agent (인허가 에이전트)
```
역할: 인허가 절차, 협의, 의제처리
지식: 인허가 의제 법령, 협의 기관 목록, 절차 흐름
LLM:  하이브리드

업무:
  - 사업별 필요 인허가 목록 생성
  - 협의 기관/근거 법령 매핑
  - 인허가 서류 체크리스트
  - 의제처리 가능 여부 판단
```

#### 💰 Accounting Agent (회계 에이전트)
```
역할: 예산, 계약, 원가, 정산
지식: 국가/지방계약법, 원가계산기준, 예정가격 작성기준
LLM:  하이브리드

업무:
  - 수의계약/경쟁입찰 기준 안내
  - 예정가격 산정 방법
  - 설계변경 시 계약금액 조정
  - 선금/중간/준공 기성 정산
```

#### 📊 Audit Agent (감사 에이전트)
```
역할: 감사 사례 분석, 리스크 사전 식별
지식: 감사원/자체 감사 사례집, 지적 패턴, 처분 기준
LLM:  하이브리드

업무:
  - 유사 감사 사례 검색
  - 현재 업무의 감사 리스크 식별
  - "이 부분 감사에서 자주 지적됩니다" 경고
  - 감사 대비 체크리스트
```

#### 📅 Task Agent (일정 에이전트)
```
역할: 할일/일정 관리, 마감 추적
지식: 없음 (범용)
LLM:  로컬 (일정 데이터 = 민감)

업무:
  - 할일 등록/조회/수정
  - 마감 알림
  - 업무 우선순위 제안
  - "오늘 뭐 해야 해?" 응답
```

#### 🧠 Profile Agent (개인화 에이전트)
```
역할: 사용자 성향 학습, 개인화 적용
지식: 사용자 프로파일 DB
LLM:  로컬 (개인 데이터 = 민감)
상태: 백그라운드 상시 관찰

업무:
  - 문체/선호 패턴 학습
  - 자주 쓰는 표현 기록
  - 업무 패턴 분석
  - 다른 에이전트에게 개인화 정보 제공
  - 사용자 프로파일 투명 공개 및 통제권 보장
```

## Agent Structure (각 에이전트의 내부 구조)

```
모든 도메인 에이전트는 동일한 구조:

domain-agent/
├── manifest.json          # 에이전트 메타정보
│   {
│     "id": "civil-engineering",
│     "name": "토목 에이전트",
│     "icon": "🏗️",
│     "version": "1.0.0",
│     "triggers": ["토목", "공사", "착공", "준공", "설계변경"],
│     "llm_preference": "hybrid",
│     "sensitivity": "internal"
│   }
├── role_prompt.txt        # 에이전트 역할 정의 프롬프트
├── rules/                 # 규칙 엔진 (LLM 불필요)
│   ├── checklists.json    # 체크리스트
│   ├── formats.json       # 서식/형식 규칙
│   └── patterns.json      # 오류 패턴
├── corpus/                # 검색 코퍼스
│   ├── laws.json          # 관련 법령 목록
│   ├── interpretations/   # 해석례
│   └── index/             # 검색 인덱스 (Whoosh/FAISS)
├── templates/             # 문서 템플릿
└── terminology/           # 전문용어 사전
```

## Execution Order

```
Phase A — Foundation (현재 ~ )
  1. 행정 에이전트 (S1 공문닥터 기반, Phase 3 진행 중)
  2. 코디네이터 (키워드 기반 라우터)
  3. 보안감사관 (정규식 PII + Gate 1,4)
  4. UI Shell (트레이 앱 + 채팅 창)
  5. LLM Router (로컬 or API key 2택)
  ═══════════════════════════════════
  → BMVP 출시: "보안 지키는 공문서 교정 AI 비서"

Phase B — Core Agents
  6. 법무 에이전트 (P2 규정레이다 기반)
  7. 일정 에이전트
  8. 개인화 에이전트 (기본 학습)
  9. 보안감사관 강화 (NER + Gate 2,3)
  10. 규칙 엔진 3-tier

Phase C — LLM Enhancement
  11. M0 civil-ai-ko 파인튜닝
  12. 코디네이터 → LLM 기반 의도 분류 업그레이드
  13. 행정/법무 에이전트 LLM 강화
  14. 하이브리드 자동 분기 (모드 C)
  15. OAuth/API key 외부 LLM 연동

  16. 토목 에이전트
  17. 건축 에이전트
  18. 인허가 에이전트
  19. 회계 에이전트
  20. 감사 에이전트

Phase E — Advanced
  21. 도메인 팩 마켓플레이스
  22. 고급 개인화 (패턴 예측)
  23. 팀 내 비서 간 연동 (같은 기관, 로컬 네트워크)
```

## BMVP (Bare Minimum Viable Product)

Phase A 완료 시 출시 가능한 최소 제품:

```
포함:
  ✅ 시스템 트레이 + 채팅 창
  ✅ 행정 에이전트 (공문서 교정, 규칙 기반)
  ✅ 코디네이터 (단순 라우터)
  ✅ 보안감사관 (정규식 PII 탐지 + 감사 로그)
  ✅ LLM Router (로컬 civil-ai-ko 또는 사용자 API key)
  ✅ 한글 프로그램 연동 (Ctrl+Shift+G)
  ✅ 로컬 데이터 저장 (암호화 SQLite)

미포함 (이후 업데이트):
  ⬜ 법무/토목/건축/인허가/회계/감사 에이전트
  ⬜ NER 기반 고급 PII 탐지
  ⬜ 개인화 학습
  ⬜ 일정 관리
  ⬜ 하이브리드 자동 분기
```

이것만으로도:
- "공문서 붙여넣기 → 교정 + 근거" 가능
- 로컬 LLM으로 윤문 가능
- 외부 LLM 연결 시 PII 마스킹 가능
- 한글 프로그램에서 단축키로 호출 가능

## Key Design Decisions (v5 Final)

1. **에이전트는 대화하지 않는다**
   입력 → 처리 → 결과. 단방향 파이프라인.
   에이전트 간 피드백 루프 없음.

2. **보안감사관은 모든 것을 본다**
   입력(Gate1) → 전송(Gate2) → 결과(Gate3) → 로그(Gate4)
   다른 에이전트는 보안감사관을 우회할 수 없음.

3. **에이전트 = 도메인 팩**
   전문 에이전트 추가 = .zip 파일 설치.
   제거 = 폴더 삭제. 핫플러그.

4. **물리적으로는 1개 LLM**
   에이전트 7개가 LLM 7개를 돌리는 게 아님.
   1개 LLM 인스턴스에 에이전트별 다른 프롬프트.
   규칙 기반 작업은 LLM을 아예 안 거침.

5. **서버 없음**
   사용자 PC에 전부 설치. 전부 로컬 저장.
   팀 동기화가 필요하면 Phase E에서 로컬 네트워크 기반으로.

6. **사용자가 LLM을 선택**
   무료(로컬) or 유료(자기 계정).
   하이브리드(민감→로컬, 안전→외부) 자동 분기.
   비용은 사용자 부담. 제품은 무료.

7. **BMVP 먼저**
   행정 에이전트 + 보안감사관 + 코디네이터.
   이것만으로 출시. 나머지는 업데이트.
